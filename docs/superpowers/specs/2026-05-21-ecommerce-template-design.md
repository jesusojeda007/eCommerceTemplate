# E-commerce Template вЂ” Design Spec
**Date:** 2026-05-21

## Overview

A Next.js + TypeScript e-commerce template designed to be cloned and customized per client. All client-specific configuration lives in a single `client.config.ts` file. The database is always PostgreSQL running in Docker, managed via Prisma. Seed data populates the DB for local development. Payment and checkout logic is abstracted so any provider can be connected without touching core logic.

---

## Stack

| Layer | Choice | Reason |
|---|---|---|
| Framework | Next.js 14+ (App Router) | SSR/SSG, Vercel-ready, industry standard |
| Language | TypeScript | Type safety across data contracts |
| Styling | Tailwind CSS + shadcn/ui | Fast to customize per client via CSS variables |
| Cart state | Zustand | Lightweight, no boilerplate |
| Auth | NextAuth.js (credentials) | Easy to swap providers per client |
| Database | PostgreSQL (Docker) | Consistent across all client projects |
| ORM | Prisma | Typed queries, auto migrations, great DX with Next.js |

## `client.config.ts`

All client customization happens here. No other file should need to change for a standard client setup.

```typescript
const config = {
  // Brand
  name: "Mi Tienda",
  tagline: "Los mejores productos",
  logo: "/client/logo.png",
  favicon: "/client/favicon.ico",

  // Theme вЂ” maps to shadcn CSS variables
  theme: {
    primary: "#16a34a",
    background: "#ffffff",
    font: "Inter",
  },

  // Contact
  whatsapp: {
    number: "+521234567890",
    messageTemplate: "Hola! Quiero pedir:\n{items}\nTotal: {total}",
  },

  // Checkout mode
  checkout: {
    mode: "whatsapp" as "whatsapp" | "qr" | "payment",
    qr: {
      imageUrl: "/client/qr-pago.png",
      instructions: "Escanea y envГ­a el comprobante por WhatsApp",
    },
  },

  // Discounts
  discounts: {
    couponsEnabled: true,
    volumeDiscountsEnabled: true,
  },

  // Feature flags
  features: {
    auth: true,
    wishlist: false,
    reviews: false,
    searchBar: true,
  },
}

export default config
```

## Database

PostgreSQL always runs in Docker. The same `docker-compose.yml` is used across all client projects вЂ” only the DB name and credentials in `.env` change per client.

```yaml
# docker-compose.yml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
```

Prisma manages the schema and migrations. `prisma/seed.ts` populates the DB with sample products, categories, and coupons for local development.

**Prisma schema models:** `Product`, `Category`, `Coupon`, `VolumeDiscount`, `User`, `Order`, `OrderItem`.

```prisma
model Product {
  id               String            @id @default(cuid())
  slug             String            @unique
  name             String
  description      String
  images           String[]
  price            Decimal
  compareAtPrice   Decimal?
  stock            Int
  category         Category          @relation(fields: [categoryId], references: [id])
  categoryId       String
  volumeDiscounts  VolumeDiscount[]
}

model Coupon {
  code             String    @id
  type             String    // "percent" | "fixed"
  value            Decimal
  minOrderAmount   Decimal?
  expiresAt        DateTime?
  active           Boolean   @default(true)
}
```

## Data Access Layer

Functions in `src/lib/data/` wrap Prisma calls and return typed app-level objects. Components never import Prisma directly вЂ” always go through these functions.

```typescript
// src/lib/data/products.ts
getProducts(filters?: ProductFilters): Promise<Product[]>
getProduct(slug: string): Promise<Product | null>

// src/lib/data/categories.ts
getCategories(): Promise<Category[]>

// src/lib/data/coupons.ts
validateCoupon(code: string): Promise<Coupon | null>
```

## Checkout Abstraction

Checkout is a plugin system. New payment methods are added by implementing `CheckoutAdapter`.

```typescript
// src/lib/checkout/types.ts
interface CheckoutAdapter {
  name: string
  handleCheckout(cart: CartItem[], customer?: Customer): Promise<void>
}
```

**Included adapters:**
- `whatsapp.ts` вЂ” formats cart as text and opens `wa.me/` link
- `qr.ts` вЂ” displays QR image + instructions modal
- `placeholder.ts` вЂ” shows "payment coming soon" UI, ready to wire up custom platform

**Future adapters (just create the file, update config):**
- `stripe.ts`
- `mercadopago.ts`
- `custom-platform.ts` вЂ” the owner's own payment platform

## Discount System

Three discount types, all resolved at cart level:

| Type | Where configured | How it works |
|---|---|---|
| Precio tachado | Product data (`compareAtPrice`) | Displayed on ProductCard and ProductDetail |
| CupГіn (% o monto fijo) | Mock coupons data / backend | User enters code in cart, validated via DataAdapter |
| Descuento por volumen | Product data (`volumeDiscounts`) | Applied automatically in cart when qty threshold is met |

Cart summary always shows: subtotal в†’ coupon discount в†’ volume discount в†’ **total**.

## Auth

NextAuth.js with credentials provider (email + password, validated against the `User` table in PostgreSQL via Prisma). Per client, swap for:
- Google / GitHub OAuth
- Magic link
- Their own auth backend

## CLAUDE.md (AI Guide)

A `CLAUDE.md` in the repo root tells any AI assistant how to work with this template:

- What `client.config.ts` controls and how to edit it
- How to start the DB: `docker compose up -d` then `npx prisma migrate dev`
- Which files are safe to touch vs. core contracts to leave alone (`prisma/schema.prisma`, `src/lib/data/types.ts`, `src/lib/checkout/types.ts`)
- How to add a new checkout adapter
- How to add a new Prisma model and expose it via the data layer
- Common tasks: change brand colors, add seed products, activate a feature flag, add a coupon
